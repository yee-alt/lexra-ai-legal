The Core Problem

LLMs Are Stateless.
That's a Feature вЂ” and a Bug.

Every API call to a large language model is, by design, a fresh computation. The model has no memory of your last message, your user's preferences, or what happened three conversations ago. This is great for scalability. It's terrible for building agents that are supposed to learn, personalize, and improve over time.

There are two root problems that every serious agent engineer must solve:

What We Need
  • Remember past conversations
  • Track user preferences
  • Learn from failures
  • Maintain consistency across sessions
  • Access external knowledge
Default LLM Behavior
  • Each call is stateless
  • No persistent user model
  • No feedback loop
  • Context window only
  • Frozen training knowledge

"The difference between a chatbot and an agent isn't reasoning вЂ” it's memory. Memory is what makes behavior compoundable."

The solution is a structured memory architecture. Not just a conversation buffer, but a layered system inspired by how human cognition actually works вЂ” drawing from decades of cognitive science research on how the brain stores, retrieves, and forgets information.

Cognitive Science Foundation

The Human Brain Has 4 Memory Systems. So Should Your Agent.

Atkinson & Shiffrin's multi-store model of human memory (1968) remains one of the most influential frameworks in cognitive psychology. It describes memory not as a single system but as a hierarchy of specialized stores вЂ” each with different capacity, duration, and function.

When we map these to the challenges of agent engineering, the correspondence is surprisingly direct:

Human в†’ Agent Memory Mapping
Human Memory Duration Agent Equivalent Storage Backend
Sensory Memory 0.5вЂ“3 sec Input preprocessing In-flight buffer
Working Memory 15вЂ“30 sec Session context window In-memory + TTL
Episodic Memory Days в†’ years Interaction history SQLite + Qdrant
Semantic Memory Permanent Knowledge & concepts Neo4j + Qdrant
Procedural Memory Permanent Skills / tool usage Fine-tuning / RLHF

The key insight: different information has different lifespans and access patterns. Mixing everything into one flat store (like a naive conversation buffer) leads to slow retrieval, noise, and missed context. Specialized stores with intelligent routing is the production-grade approach.

Notice that all four scoring formulas share a common structure: (semantic_relevance Г— weights) Г— importance_multiplier. The importance multiplier is always in the range [0.8, 1.2] вЂ” deliberately narrow. This ensures importance adjusts the ranking rather than dominating it. A highly important but irrelevant memory should never outrank a highly relevant one.

Code: The MemoryTool API

A Unified Interface Over All 4 Layers

In production, you don't want agents calling four different memory systems with four different APIs. The MemoryTool provides a single execute(action, **kwargs) interface. Here are the most important operations:

1. Adding Memories

Every memory type gets the same "add" action. The memory_type parameter routes to the right backend:

2. Searching Across Memory

Retrieval supports single-type or multi-type queries, with a minimum importance filter to cut noise:

memory_tool.py вЂ” search operations 
# Basic cross-memory search
results = memory_tool.execute("search",
    query="Python learning progress",
    limit=5
)

# Scoped to a single memory type
results = memory_tool.execute("search",
    query="user milestones",
    memory_type="episodic",   # only look in episodic
    limit=3
)

# Multi-type with importance filter (removes noise)
results = memory_tool.execute("search",
    query="function definition syntax",
    memory_types=["semantic", "episodic"],
    min_importance=0.5          # skip low-value memories
)

The Most Important Mechanic

Consolidation: Promoting Short-Term to Long-Term

This is where the cognitive science analogy becomes engineering gold. In humans, sleep is when the hippocampus replays the day's experiences and consolidates important ones into the neocortex for long-term storage. In your agent, consolidation is a deliberate operation you trigger вЂ” or schedule вЂ” to do the same thing.

Memory Consolidation Flow

STEP 01
Accumulate
Events fill working memory during session
STEP 02
Score
Each item evaluated: importance Г— frequency
STEP 03
Promote
Items above threshold в†’ episodic or semantic
STEP 04
Forget
Low-importance items TTL-expired or pruned

memory_tool.execute("consolidate",
    from_type="working",
    to_type="episodic",
    importance_threshold=0.7   # only what matters survives
)

# Distill repeated episodic patterns в†’ semantic (weekly job)
memory_tool.execute("consolidate",
    from_type="episodic",
    to_type="semantic",
    importance_threshold=0.8
)

# в”Ђв”Ђ 3 forgetting strategies в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ

# Strategy 1: Prune low-importance junk

    strategy="importance_based", threshold=0.2)

# Strategy 2: Time-based cleanup (stale context)

    strategy="time_based", max_age_days=30)

# Strategy 3: Capacity management (keep the best N)

    strategy="capacity_based", threshold=0.3)

Don't just delete from the SQL/SQLite store. If you're using a vector database (Qdrant) and a graph database (Neo4j), forgetting requires deleting from all three stores in a coordinated transaction. Partial deletes create orphaned vectors that pollute future searches and slowly degrade retrieval quality.

Advanced: Semantic Memory

Knowledge Graphs Change What's Possible

Most people stop at vector search. Semantic memory in a mature system goes further: it builds a knowledge graph as a side effect of storing memories. When you add "Zhang San is a Python developer who works on ML pipelines," the system extracts:

Auto-extracted Knowledge Graph Entities & Relations
Entity Type Relation Target
Zhang San PERSON IS_A Python Developer
Zhang San PERSON WORKS_ON ML Pipelines
ML Pipelines CONCEPT RELATED_TO Python
Python Developer ROLE USES Python Language

Now when a user asks "What tools does Zhang San use?" вЂ” the graph traversal finds the answer through two hops, even if those exact words were never stored together as a single memory. This is the key power vector search alone can never provide.

The combined retrieval score reflects this dual architecture:

Semantic Memory Score = (vector_similarity Г— 0.7 + graph_proximity Г— 0.3) Г— (0.8 + importance Г— 0.4)

The 70/30 split between vector and graph is intentional: vector search handles semantic similarity well for direct queries; graph search shines for indirect, relational queries. Together they cover what neither does alone.

Putting It Together

Wiring It Into Your Agent in 10 Lines

The entire memory system is exposed as a MemoryTool that slots into any agent's tool registry. No need to touch the agent class itself:

app.py вЂ” full agent setup with memory 
from hello_agents import SimpleAgent, HelloAgentsLLM, ToolRegistry
from hello_agents.tools import MemoryTool

llm = HelloAgentsLLM()
agent = SimpleAgent(name="Memory Agent", llm=llm)

# One line to add all 4 memory layers
memory_tool = MemoryTool(
    user_id="user_42",              # isolates this user's memories
    memory_types=[                   # choose which layers to enable
        "working", "episodic", "semantic"
    ]
)

registry = ToolRegistry()
registry.register_tool(memory_tool)
agent.tool_registry = registry

# Now the agent can remember across sessions
agent.run("Remember: I prefer Python examples over pseudocode")
# в†’ stores to semantic memory, importance auto-scored

agent.run("How do I write a decorator?")
# в†’ retrieves user preference, formats response accordingly

The user_id parameter is your isolation mechanism. All four memory backends (in-memory store, SQLite, Qdrant, Neo4j) are namespaced by user_id. This means you can run a single agent serving thousands of users, each with their own fully isolated memory state вЂ” without any application-level partitioning logic.

Key Takeaways

What You Should Walk Away Knowing

1. Don't conflate context window with memory. The context window is not memory вЂ” it's the current computation. Memory is what persists between computations. If you're only passing chat history into the context, you're not building memory, you're building a longer prompt.

2. Each memory type serves a different access pattern. Working memory is for speed (current task). Episodic is for events (what happened). Semantic is for knowledge (what's true). Perceptual is for modalities (what was seen/heard). Using the wrong layer for the wrong data creates retrieval noise.

3. The importance score is not optional. Every memory should carry an importance float in [0.0, 1.0]. This single number enables forgetting, consolidation, and noise filtering вЂ” the three operations that keep your memory system from becoming a garbage dump over time.

4. Consolidation is scheduled work, not reactive work. Build consolidation triggers into your agent's lifecycle: on session end, daily, or after N interactions. Treat it like a database maintenance job, not a user-facing feature.

5. Vector search + graph traversal > vector search alone. For any memory system that needs to reason about relationships (not just similarity), a knowledge graph backend pays for its complexity within weeks of deployment.
